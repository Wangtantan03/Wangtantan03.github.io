# 测试的基本流程

作为软件测试实习生，在某传统公司的科技研发部工作有一周的时间了，部门主要做“互联网+演艺”的B2B产品。最近一周跟着团队做了冒烟测试和功能测试，对测试的基本流程有了清晰的认识，特此来记录一下我对测试的基本流程的理解。

# 具体内容

# 需求了解和评审阶段

该阶段主要可以分为两个部分：

  • 参与需求的初评：根据prd文档或者原型的初稿，评估需求是否可以实现、主逻辑是否正确以及是否满足产品预期，并提出相应的修改意见。
  • 参与需求的评审：主要是为了更加深入的了解需求，验证初评时的修改点。

# 制定测试计划阶段

测试计划的制定一般由测试负责人制定，该阶段需要明确两个时间点：

明确两个时间点：

1、产品经理需要给定计划发布的时间； 2、开发负责人需要给定计划提测的时间。

# 测试用例设计阶段

参与测试的人员根据产品prd文档和产品原型图，以及开发人员的设计方案等其他可利用的资源，利用xmind工具进行测试点的提取，然后进行测试用例的编写， 最后通过禅道系统对编写的测试用例进行管理。

# 开发提测阶段（冒烟测试阶段）

开发提测阶段也就是冒烟测试阶段，该阶段的目的是确保提测版本的基本功能和实现的逻辑均符合需求，同时可以将后续正式测试工作中可能存在的未知风险 降低到最低，防止存在的bug导致测试进度受到阻碍，我理解的冒烟测试最大的优点就是可以节约测试的时间成本，减少测试的轮数。

冒烟测试的定义：

冒烟测试由开发人员和测试人员共同进行，是一种版本验证测试，主要是确认新的版本是否存在致命的bug,功能可以正常运行（即不会出现跑不通的情况），不 会影响下一轮的测试。 如果符合上述情况，该版本就可以进入下一轮测试

# 测试执行阶段

该阶段首先根据提测的输入，成功部署测试环境后，测试人员根据测试用例对经过冒烟测试后的版本和后续bug修复后的版本进行测试，并将bug统一记录在bug 管理系统中（比如：禅道等），最后测试结束后，出一份测试总结报告交付给项目团队。

需要注意的几点情况：

  1. 在测试的过程中，对不正确的用例进行修改或补充，提高测试用例的覆盖率；同时反思自己设计测试用例时为什么没有想到。
  2. 对于存在争议的bug，测试人员应该主动和产品经理去沟通，如果沟通之后还是无法达成一致，可以同开发负责人、产品负责人等讨论后再做最终的决定。

# 上线阶段

在测试环境测试通过之后，提交预发布申请，部署系统到预发布环境中进行测试，同时提醒产品经理做相应的验收工作； 通过之后进行打包发布，并进行线上的回归测试，若线上回归测试通过即发布完毕。 该阶段完成之后，测试人员进行自我测试的总结，最好出一份自我测试总结的报告
